# 구현 계획서 - 성능 최적화, 맞춤형 타임프레임 탐색, 시작 화면 UX 개선 및 API 키 동기화 버그 해결

본 계획서는 사용자가 제안한 4가지 사항을 해결하기 위한 구체적인 기술적 아키텍처 및 구현 방안을 정리한 문서입니다.

---

## 1. 서브 김프 데이터 활성화 시 차트 렉 해결 (성능 최적화)

### 원인 분석
1. **과도한 실시간 틱 처리:** 업비트/빗썸 웹소켓 연결이 활성화되어 있을 때, 파트너 거래소의 실시간 틱 데이터가 수신될 때마다 동기식으로 `updateRealtimeKimchi`가 호출되고 있습니다. 이로 인해 초당 수십 회의 차트(`.update()`) 및 DOM 강제 리페인트가 발생하여 렌더링 프레임 드랍(렉)을 유발합니다.
2. **반복적인 O(N) 순회:** 마우스 크로스헤어가 차트 내부에서 움직일 때마다 `store.mainData`, `store.volumeData`, `store.kimchiData`에 대해 `.find()` 메서드를 사용한 선형 탐색이 매 프레임 실행됩니다. 데이터 크기가 수백 개 이상일 때 이 연산은 심각한 CPU 오버헤드를 발생시킵니다.
3. **가로축 스케일 연산 병목:** 틱마다 `window.syncPriceScaleWidths()`가 호출되어 불필요한 레이아웃 리플로우(Reflow)를 반복적으로 유발합니다.

### 개선 방안
- **웹소켓 틱 업데이트 디바운스/쓰로틀링:** 실시간 김프 틱 수신 시 즉시 화면을 갱신하지 않고, `requestAnimationFrame`과 시간차 비교를 이용해 최소 120ms 간격으로 웹소켓 차트/DOM 갱신을 제한하는 쓰로틀링(Throttler)을 적용합니다.
- **O(1) Map 구조 기반 고속 매핑:**
  - `store` 전역 객체 내에 `store.mainDataMap`, `store.volumeDataMap`, `store.kimchiDataMap` 해시맵을 추가합니다.
  - 차트에 신규 데이터를 세팅(`setData`)할 때 해당 맵을 일괄 업데이트합니다.
  - 크로스헤어 무브 핸들러에서는 `.find()` 대신 `Map.prototype.get()`을 사용하여 $O(1)$ 복잡도로 데이터를 단시간에 가져옵니다.
  - Ticker 정보 검색 시에도 `store.tickerRowMap`을 활용해 테이블 선형 탐색 빈도를 최소화합니다.

## 2. 타임프레임 좌우측 방향키 탐색 범위 커스텀화

### 원인 분석
- `static/_main.js`의 `keydown` 이벤트 핸들러에서 좌우 방향키 입력 시 하드코딩된 전체 타임프레임 배열(`tfArray`)을 기준으로 이전/다음 타임프레임을 탐색합니다.
- 이로 인해 사용자가 설정 화면에서 원하지 않는 타임프레임을 비활성화(숨김) 처리했음에도 방향키 이동 시 숨겨진 타임프레임까지 순회하는 오작동이 발생합니다.

### 개선 방안
- 키다운 핸들러 내에서 `tfArray`를 전역 `store.visibleTfs` (또는 로컬 스토리지에 저장된 유저 커스텀 설정)의 우선순위를 반영하도록 동적으로 수정합니다.
- 커스텀 설정이 누락된 경우에만 기존의 전체 타임프레임 목록을 폴백(Fallback)으로 적용하도록 안정성을 확보합니다.

## 3. 시작 화면(Start Screen) 프리미엄 UX/UI 개선 (코인 로고 & 글래스모피즘)

### 디자인 컨셉
- **다이내믹 백그라운드:** CoinMarketCap의 인기 코인 UID를 기반으로 이미지 로고(`s2.coinmarketcap.com`)를 가져와 화면 배경에 부유하는 유리 방울(Floating Coin Glass Bubbles) 형태로 동적 배치합니다.
- **인터랙티브 패러랙스:** 사용자의 마우스 움직임에 반응하여 배경의 코인 방울들이 서로 다른 레이어 속도로 움직이는 3D 입체 미러링 효과를 제공합니다.
- **글래스모피즘 입력 카드:** 중앙 입력 카드를 `backdrop-filter: blur(25px)` 및 미세한 반사 테두리를 통해 우주적이고 세련된 디스플레이 카드로 업그레이드합니다.

### 구현 세부사항
- `static/start.js`에서 시작 화면 렌더링 시 HTML 내에 `#start-background` 요소를 확장하고, JS를 통해 코인 방울들을 임의의 위치, 애니메이션 속도, 투명도로 동적 생성합니다.
- 화면 내 마우스 이동(`mousemove`) 시 각 방울들의 CSS `transform` 속성을 조절하여 고성능의 패러랙스 인터랙션을 구현합니다.

## 4. 배포 서버에서 CMC API Key 설정 업데이트 오류 해결

### 원인 분석
1. **휘발성/읽기 전용 파일 시스템:** 배포 환경(Railway 등)은 컨테이너가 다시 켜지거나 파일 시스템이 읽기 전용(Read-only)으로 마운트되는 경우가 많아 `.env` 파일에 쓰기 작업을 시도하면 권한 에러(PermissionError)가 발생하여 전체 프로세스가 예외 중단될 수 있습니다.
2. **다중 프로세스 메모리 격리:** Gunicorn/Uvicorn이 다수의 워커(Worker) 프로세스로 구동될 때, 특정 워커 메모리의 `config.CMC_API_KEY`만 업데이트되어 다른 워커가 처리하는 시세 수집 요청에는 업데이트된 API 키가 전달되지 않습니다.

### 개선 방안
- **클라이언트 전송 기반 Stateless 키 보관:**
  - 사용자의 API 키는 브라우저 `localStorage`에 계속 안전하게 저장되어 있습니다.
  - 프론트엔드에서 `/api/market-data` 및 `/api/market-data-silent` 등을 호출할 때 헤더에 `X-CMC-API-KEY` 항목으로 저장된 키를 실어 보냅니다.
  - 백엔드(FastAPI)에서는 요청 수신 시 헤더에 `X-CMC-API-KEY`가 존재할 경우 메모리상의 `config.CMC_API_KEY`를 해당 값으로 즉시 동적 셋업하도록 설계합니다.
  - 이를 통해 서버 파일 시스템 손상 우려가 전혀 없고, 여러 워커 인스턴스 간에도 요청 단위로 정확한 API 키가 동기화되어 시총 데이터가 정상 작동합니다.
- **예외 방어:** `config.py`의 파일 쓰기 함수(`set_cmc_api_key`)를 `try...except`로 감싸 읽기 전용 환경에서도 서버가 에러 없이 구동되도록 가드(Guard)를 세웁니다.

## 검증 계획

### 자동 테스트
- `run_command`를 사용하여 백엔드 FastAPI 서버를 구동하고, 새로운 `X-CMC-API-KEY` 헤더 필터를 포함한 모의 요청(curl)을 보내 정상적으로 CMC API 키를 동적 적용 및 조회하는지 확인합니다.

### 수동 검증
1. 로컬 브라우저 구동 후 시작 화면에서 32글자의 임의 키를 입력하거나 스킵하여 대시보드에 정상 진입하는지 확인합니다.
2. 마우스 스크롤 및 마우스 오버(크로스헤어 무브) 시 UI가 프레임 드랍 없이 부드럽게 작동하는지 브라우저 F12 렌더링 성능 검사로 검증합니다.
3. 설정 모달에서 API 키를 업데이트했을 때, 즉시 시가총액 정보가 갱신되는지 네트워크 탭을 확인합니다.
4. 키보드 방향키(좌우)를 조작하여 숨긴 타임프레임이 건너뛰어지고 표시된 타임프레임으로만 퀵 스위칭되는지 검증합니다.
