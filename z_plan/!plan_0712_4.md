# [UX 성능 업그레이드] Service Worker + Cache API 기반 클라이언트 전용 캐싱 시스템

## 배경 및 목표

현재 `localStorage`를 활용한 수동 캐싱 방식의 문제점:
- `JSON.parse(800개 코인)` 동작이 **메인 스레드를 블로킹** (동기 동작)
- `localStorage` **5MB 저장 한계** (데이터가 크면 캐시 실패)
- 브라우저 내장 HTTP 캐시와 **중복 운영** 비효율

**목표**: Service Worker + Cache API로 전환하여 `/api/market-data` HTTP 응답 자체를 브라우저 내부에 캐싱합니다.
- 백엔드 부하 **0%**
- 메인 스레드 블로킹 **0ms** (비동기 전용 스레드)
- 업비트·바낸과 동일한 **Stale-While-Revalidate** 전략

---

## 동작 원리 (Stale-While-Revalidate)

```
새로고침 시:
1. [0ms]      Service Worker가 /api/market-data 요청을 가로챔
2. [~5ms]     Cache API에서 캐시된 응답 즉시 반환 → 화면에 Top 30 바로 렌더링
3. [백그라운드] 서버에 실제 요청 → 응답 도착 시 Cache 갱신 + 800개 전체 재렌더링
```

## 개방 질문 (검토 필요)

> [!IMPORTANT]
> **Q1. `/sw.js` 루트 경로 서빙 가능 여부**
> Service Worker는 반드시 루트 경로(`/sw.js`)에서 서빙되어야 합니다.
> 현재 `app.py`의 `StaticFiles` 마운트 구조에서 `/sw.js`를 루트에서 직접 서빙 가능한지 확인이 필요합니다.
> (간단히 `@app.get("/sw.js")` 라우터 추가로 해결 가능합니다.)

> [!IMPORTANT]
> **Q2. 현재 localStorage 방식 제거 여부**
> Service Worker 전환 후 기존 `localStorage` 기반 캐시 코드(`sellnance_market_data_cache`)를 **완전히 제거**할지, 구형 브라우저를 위한 폴백으로 **병행 유지**할지 결정이 필요합니다.

> [!NOTE]
> **Q3. `/api/market-map` 추가 캐싱 여부**
> 코인 맵핑 정보(`/api/market-map`)도 같은 SW 캐시 전략에 포함하면 초기 로딩을 추가로 단축할 수 있습니다.
