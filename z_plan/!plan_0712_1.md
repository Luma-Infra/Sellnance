# CoinMarketCap API 이원화 캐싱 시스템 구현 계획서

이 계획서는 CoinMarketCap(CMC) API 호출로 인한 크레딧(토큰) 소모를 원천 차단하고 유저 경험을 개선하기 위한 이원화 캐싱 메커니즘을 수립합니다.

1. **사장님 키 (서버 전용)**: **24시간(하루 1회)** 주기로 캐싱하며, 레일웨이(Railway) 서버 재부팅 시 크레딧이 낭비되는 것을 막기 위해 로컬 파일에 영구 저장(JSON)합니다.
2. **유저 개별 키 (브라우저 로컬 저장)**: API 키 해시값 기준으로 메모리에 **15분** 동안만 임시 캐싱하여 파워 유저들이 각자의 크레딧을 소모해 최신 시총 데이터를 실시간으로 볼 수 있도록 지원합니다.

## 제안된 변경 사항

### [컴포넌트] 백엔드 캐시 레이어

#### [수정] [api_manager.py](file:///c:/Users/78831/Sellnance/modules/api_manager.py)
* 개별 캐시 쿨타임 변수 정의:
  - `OWNER_CACHE_TIMEOUT = 86400` (24시간)
  - `USER_CACHE_TIMEOUT = 900` (15분)
* 사장님 전용 키 캐시는 서버 재빌드/재부팅 시 유실되지 않도록 `static/cmc_owner_cache.json` 파일에 쓰기/읽기(Persistent Storage) 로직 추가.
* 유저 개별 키 캐시는 서로 간섭하지 않도록 메모리 딕셔너리(`USER_CMC_CACHES = {}`)에 키 해시별로 분리 저장.
* `get_cached_data` 함수가 유저가 보낸 API 키 매개변수를 직접 받아 쿨타임 정책 분기 처리하도록 수정.

#### [수정] [app.py](file:///c:/Users/78831/Sellnance/modules/app.py)
* `/api/market-data` 라우터가 헤더의 `X-CMC-API-KEY`를 직접 파싱하여 `api_manager.get_cached_data()` 인자로 전달하도록 수정 (전역 `config` 변수를 오염시켜 발생하는 레이스 컨디션 차단).

## 검증 계획

### 수동 검증
1. 로컬에서 서버 가동 후 첫 호출 시 사장님 기본 키로 데이터를 가져오고 `static/cmc_owner_cache.json` 파일이 정상 생성되는지 확인.
2. 로컬 서버를 강제 재부팅한 후 재요청 시, API를 재호출하지 않고 저장된 파일 캐시를 읽어와 0.001초 만에 응답을 뱉는지 검사.
3. 개인 API 키를 등록한 유저로 요청 시 15분 주기로 갱신 타이머가 정상 카운트다운되는지 확인.
