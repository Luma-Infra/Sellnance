# 소켓 가격 업데이트 분석 및 개선 사항

## 🔍 소켓 가격 변화 부자연스러움 원인 분석

### 구조 설명
현재 시스템의 가격 업데이트 경로는 두 개의 레이어로 구성됩니다.

```
WebSocket 수신 → renderRealtimeRow (즉시 DOM 업데이트)
               ↘
               tickerBuffer 누적 → 10초마다 배치 처리
```

### 문제 1: 칼군무 (일괄 갱신 어색함)
**원인**: 바이낸스 WebSocket `!ticker@arr`는 **전체 코인 가격이 한 번에 배열로** 옵니다. 이것이 `forEach`로 한꺼번에 `requestAnimationFrame` 큐에 쌓이면 브라우저가 **같은 프레임에 모든 DOM을 동시에 갱신**하여 칼군무처럼 보입니다.

**개선**: 각 코인마다 `0~600ms`의 개별 랜덤 딜레이를 부여하여 자연스럽게 흩어지도록 처리했습니다.

### 문제 2: 첫 시작 후 시세 미표시
**원인**: `lastRenderMap`에 등록된 코인이 없는 경우에도 `now - lastRender < 800`이 0ms이므로 항상 통과하지만, `visibleSymbols`에 코인이 없어 무시됩니다. 테이블이 완전히 렌더된 이후 소켓 첫 메시지는 `isFirstRender` 여부와 무관하게 800ms 대기가 있었습니다.

**개선**: `lastRenderMap.has(tId)`가 false인 첫 렌더는 쓰로틀 없이 즉시 반영합니다.

### 문제 3: tickerBuffer 배치 인터벌 10초
**원인**: 배치 인터벌이 10초라 소켓으로 업데이트된 데이터가 `renderTable(isRealtime=true)` 통해 재정렬되기까지 최대 10초가 걸립니다.

**개선**: **3초**로 단축. (단, 성능 이슈가 있다면 5초로 조정 권장)

### 문제 4: 800ms 쓰로틀이 너무 길다
**현재 상태**: 화면에 보이는 코인도 800ms에 한 번만 DOM 업데이트 허용.
바이낸스 `!ticker@arr`는 약 1~2초 주기로 전체 데이터를 전송합니다. 800ms 쓰로틀이면 **일부 업데이트가 유실**됩니다.

**개선**: **500ms**로 단축. 이로써 바이낸스 2초 주기 소켓에서 최소 1회는 렌더링을 보장합니다.

## 💡 추가 권장 사항

### 업비트 소켓 반응성 개선 (현재 미적용)
업비트는 체결가(`trade`) 타입으로 구독하면 모든 체결 시마다 데이터가 오지만, 현재는 `ticker` 타입(1초 집계치)으로 구독 중입니다. 체결 즉시 반영을 원한다면 `trade` 타입으로 변경을 고려해볼 수 있습니다.

### 부드러운 가격 보간 (고급)
현재 가격이 점프하는 느낌이라면, 새 가격을 받았을 때 현재가→목표가까지 ~200ms에 걸쳐 보간(lerp)하는 애니메이션을 적용하면 매우 자연스러운 흐름이 됩니다. CSS `counter`나 JS `requestAnimationFrame` 보간으로 구현 가능합니다.
