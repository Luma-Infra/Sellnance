# Implementation Plan - chart_data.js 및 builder.py 정교한 모듈화 및 슬림화

본 계획은 기존의 핵심 동작 로직을 100% 보존하면서, 1000줄이 넘는 프론트엔드 차트 데이터 관리 파일인 `static/chart_data.js`와 백엔드 데이터 조립 파일인 `modules/builder.py`를 정교하게 서브 모듈들로 나누어 코드 크기를 극적으로 줄이고 유지보수성을 향상하는 것을 목표로 합니다.

## User Review Required

> [!IMPORTANT]
> - **하위 호환성 100% 보장**: 호출 규격 및 기존 동작 흐름을 완벽히 유지합니다. 백엔드의 `from modules import builder` 및 프론트엔드의 `import { fetchHistory } from "./chart_data.js"` 등의 기존 인터페이스는 단 한 줄의 변경 없이 그대로 호환되도록 구성합니다.
> - **서브 모듈 분리**:
>   - `static/chart_data.js` 에서 순수 과거 김프 조립 연산을 다루는 `calculateKimchiData` 함수를 `static/chart_data_kimchi.js`로 분리합니다.
>   - `modules/builder.py` 에서 각 거래소별 개별 행 빌드를 담당하는 `build_binance_row`, `build_upbit_row`, `build_bybit_row` 함수들을 `modules/builder_binance.py`, `modules/builder_upbit.py`, `modules/builder_bybit.py`로 각각 완전히 분리합니다.
